# What is React?

**React** is an open-source JavaScript library used to build **User Interfaces (UI)**, especially for web applications.

It was developed by Meta (formerly Facebook) and is currently one of the most popular frontend technologies.

Instead of manually manipulating HTML and the DOM, React allows developers to build applications using reusable components.

---

## Traditional JavaScript Approach

Imagine creating a Task Manager using plain JavaScript.

```html
<div id="app"></div>
```

```javascript
const app = document.getElementById("app");

const heading = document.createElement("h1");
heading.textContent = "TaskFlow";

app.appendChild(heading);
```

For small applications this is fine.

But as applications grow:

- Hundreds of DOM elements
- User interactions
- State changes
- API calls
- Complex UI updates

Managing everything manually becomes difficult.

---

## React Approach

React allows us to describe **what the UI should look like**.

```jsx
function App() {
  return <h1>TaskFlow</h1>;
}
```

React handles the DOM updates automatically.

This makes development:

- Faster
- Cleaner
- More maintainable
- Easier to scale

---

# Why React Became So Popular

React solves several problems faced in traditional web development.

## 1. Component-Based Architecture

Instead of writing one huge file, we create small reusable building blocks.

We can use it multiple times throughout the application.


React updates only the first task.

Not the entire page.

# Understanding the Virtual DOM

This is one of React's most important concepts.

---

## What is the DOM?

DOM stands for:

**Document Object Model**

When a webpage loads:

```html
<body>
  <h1>TaskFlow</h1>
</body>
```

The browser converts it into a tree structure.

```text
Document
│
└── Body
    │
    └── H1
        │
        └── TaskFlow
```

JavaScript interacts with this DOM tree.

---

## Problem with Direct DOM Manipulation

Changing the DOM is expensive.

Example:

```javascript
document.getElementById("task").innerText = "Completed";
```

For a small page it's fine.

For a large application with thousands of elements:

- Frequent DOM updates
- Slower rendering
- Performance issues

---

# What is Virtual DOM?

React creates a lightweight copy of the real DOM.

```text
Real DOM
    ↓
Virtual DOM
```

Think of it as a blueprint.

---

## Analogy: Office Whiteboard

Imagine an office team managing project tasks.

### Traditional DOM

Every time a task changes:

```text
Erase board
Rewrite board
```

Lots of work.

---

### Virtual DOM

Instead:

```text
Make changes on a draft copy
Compare with original
Update only differences
```

Much faster.

---

# How React Uses Virtual DOM

When state changes:

```jsx
setTasks(...)
```

React performs:

### Step 1

Creates a new Virtual DOM.

```text
Old Virtual DOM
```

vs

```text
New Virtual DOM
```

---

### Step 2

Compares them.

This process is called:

## Diffing

```text
Old:
□ Learn React

New:
✓ Learn React
```

---

### Step 3

Updates only the changed node.

```text
Update checkbox only
```

Not the entire page.

---

# React Rendering Flow

```text
User Action
     │
     ▼
State Changes
     │
     ▼
New Virtual DOM Created
     │
     ▼
Diffing Algorithm
     │
     ▼
Minimal DOM Updates
     │
     ▼
Screen Updated
```

❌


✅



JSX:

```jsx
<div className="container"></div>
```

# State vs Props

This is one of the most important React concepts.

---

# Props

Props are:

```text
Read Only
Passed by Parent
Cannot be changed by Child
```

Example:

```jsx
<TaskItem task="Learn React" />
```

The child receives the value.

It cannot modify it.

---

# State

State is:

```text
Component's own data
Can change over time
Triggers UI updates
```

Example:

```jsx
const [count, setCount] = useState(0);
```

State belongs to the component.

---

# Analogy

Imagine a school.

## Props

```text
Teacher
   │
   ▼
Student
```

Students receive it.

They cannot change it.

---

## State

Student maintains personal notes.

```text
Student
   │
   ▼
Notebook
```

Student can update it anytime.

---

# State vs Props Comparison

| Feature | Props | State |
|----------|--------|--------|
| Mutable | No | Yes |
| Owned By | Parent | Component |
| Passed To Child | Yes | No |
| Triggers Re-render | Yes | Yes |
| Read Only | Yes | No |

# Lifting State Up

One of the most important React concepts.

---

# The Problem

Suppose:

```text
TaskForm
TaskList
```

Both components need access to tasks.

---

# Incorrect Approach

```text
TaskForm State
TaskList State
```

Separate states.

Data becomes inconsistent.

---

# Correct Approach

Move state to common parent.

```text
App
│
├── tasks state
│
├── TaskForm
└── TaskList
```

This is called:

## Lifting State Up

---

# Before Lifting State

```text
TaskForm
│
└── tasks state

TaskList
│
└── tasks state
```

Two sources of truth.

Bad practice.

---

# After Lifting State

```text
App
│
└── tasks state
    │
    ├── TaskForm
    └── TaskList
```

One source of truth.

Recommended approach.

---

# Example

## Parent Component

```jsx
function App() {
  const [tasks, setTasks] =
    useState([]);

  return (
    <>
      <TaskForm
        tasks={tasks}
        setTasks={setTasks}
      />

      <TaskList
        tasks={tasks}
      />
    </>
  );
}
```

---

```jsx
function TaskList({ tasks }) {
  return (
    <ul>

        <li>{task}</li>

    </ul>
  );
}
```

Now both components share the same state.

```text
    ↓
```
```text
State
```

```jsx
  state,

          action.payload

      };

    default:
      return state;

  }

}
```

| Expensive Calculations | useMemo |
| Stable Functions | useCallback |

---

# Best Practices for Hooks

---

## 1. Keep Hooks at Top Level

✅

```jsx
const [tasks, setTasks] =
  useState([]);
```

---

## 2. Use Multiple useEffect Hooks

Better:

```jsx
useEffect(loadTasks, []);
useEffect(saveTasks, [tasks]);
```

instead of one giant effect.

---

## 3. Use useReducer for Complex State

Avoid:

```jsx
10+ useState Hooks
```

---

## 4. Don't Overuse useMemo

Bad:

```jsx
useMemo(() => name, [name]);
```

No performance gain.

---

## 5. Don't Overuse useCallback

Use only when needed.

---

## 6. Keep Effects Focused

One effect:

```text
Load Data
```

Another:

```text
Save Data
```

```jsx
  useState,
  useEffect
} from "react";

function TaskFlow() {

  const [tasks, setTasks] =
    useState([]);
```

# React.memo()

One of the easiest optimizations.

---

# Purpose

Prevent unnecessary component re-renders.

---

# Example Without React.memo

```jsx
function Header() {

  console.log(
    "Header Rendered"
  );

  return <h1>TaskFlow</h1>;

}
```

---

Whenever parent re-renders:

```text
Header Rendered
```

appears again.

---

# Using React.memo

```jsx
const Header = React.memo(
  function Header() {

    console.log(
      "Header Rendered"
    );

    return <h1>TaskFlow</h1>;

  }
);
```

---

# What Happens?

If props remain unchanged:

```text
No Re-render
```

---

# Analogy

Imagine printing documents.

Without memoization:

```text
Print Entire Book
Every Time
```

---

With memoization:

```text
Reuse Existing Copy
```

unless content changes.

---

# React.memo Flow

```text
Parent Re-renders
       │
       ▼
Props Changed?
   │         │
  Yes       No
   │         │
Render    Skip
```

---

# When to Use React.memo

Good Candidates:

```text
Header
Sidebar
Navigation
Task Cards
Static Components
```

---

# When NOT to Use React.memo

Avoid:

```text
Tiny Components
Components That Always Change
```

because memoization itself has a cost.

---

# useMemo()

Used to optimize expensive calculations.

---

# Problem

Suppose TaskFlow contains:

```text
10,000 Tasks
```

---

# Expensive Operation

```jsx
const completedTasks =
  tasks.filter(
    task =>
      task.completed
  );
```

---

Every render:

```text
Filter Runs Again
```

Even if tasks didn't change.

---

# Solution

```jsx
useMemo()
```

---

# Example

```jsx
const completedTasks =
  useMemo(() => {

    return tasks.filter(
      task =>
        task.completed
    );

  }, [tasks]);
```

---

# What Happens?

```text
Tasks Changed?
      │
      ▼
     Yes
      │
 Recalculate
```

---

Otherwise:

```text
Reuse Cached Result
```

---

# Visual Flow

```text
Render
   │
   ▼
Dependencies Changed?
      │
 ┌────┴────┐
 │         │
Yes       No
 │         │
Compute  Cached
```

---

# Real TaskFlow Example

Calculate statistics:

```text
Completed Tasks
Pending Tasks
High Priority Tasks
```

---

Without useMemo:

```text
Recalculate Every Render
```

---

With useMemo:

```text
Recalculate Only When Tasks Change
```

---

# useCallback()

Used to optimize functions.

---

# Problem

Functions are recreated during every render.

---

# Example

```jsx
function App() {

  const deleteTask =
    () => {};

}
```

---

Every render:

```text
New Function Created
```

---

# Why Is This a Problem?

Suppose:

```jsx
<TaskList
  onDelete={deleteTask}
/>
```

Child receives:

```text
New Function Reference
```

React thinks:

```text
Prop Changed
```

and re-renders.

---

# Solution

```jsx
useCallback()
```

---

# Example

```jsx
const deleteTask =
  useCallback(() => {

    console.log(
      "Delete Task"
    );

  }, []);
```

---

# Result

```text
Same Function Reference
```

until dependencies change.

---

# useCallback Flow

```text
Render
   │
   ▼
Dependencies Changed?
      │
 ┌────┴────┐
 │         │
Yes       No
 │         │
New Fn   Reuse Fn
```

---

# useMemo vs useCallback

| Hook | Memoizes |
|--------|-----------|
| useMemo | Values |
| useCallback | Functions |

---

# Example Comparison

### useMemo

```jsx
const completedCount =
  useMemo(
    () =>
      tasks.length,
    [tasks]
  );
```

Stores:

```text
Calculated Value
```

---

### useCallback

```jsx
const deleteTask =
  useCallback(
    () => {},
    []
  );
```

Stores:

```text
Function Reference
```


# Why Custom Hooks?

```text

```

Without custom hooks:

```text

Repeated useEffect
Repeated Logic
```

---

# Example

```jsx

    useState([]);
```

# Same Code Elsewhere

```jsx

  const [tasks, setTasks] =
    useState([]);

  const [loading,
    setLoading] =
    useState(true);

}
```

Duplicate code.

---

# Creating Custom Hook

```jsx
function useTasks() {

  const [tasks, setTasks] =
    useState([]);

  const [loading,
    setLoading] =
    useState(true);

  useEffect(() => {

    fetchTasks();

  }, []);

  return {

    tasks,
    loading

  };

}
```

---

# Using Custom Hook

```jsx

  const {
    tasks,
    loading
  } = useTasks();

}
```

---

# Dashboard Reuse

```jsx
  const {
    tasks,
    loading
  } = useTasks();

}
```

Same logic.

No duplication.

# Example: useLocalStorage

A common interview question.

---

```jsx
function useLocalStorage(
  key,
  initialValue
) {

  const [value, setValue] =
    useState(() => {

      const saved =
        localStorage.getItem(
          key
        );

      return saved
        ? JSON.parse(saved)
        : initialValue;

    });

  useEffect(() => {

    localStorage.setItem(
      key,
      JSON.stringify(value)
    );

  }, [key, value]);

  return [
    value,
    setValue
  ];

}
```

```jsx
  setTheme
] = useLocalStorage(
  "theme",
  "light"
);
```
